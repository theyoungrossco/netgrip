# 0.3 plan — clarity, terminology & teaching the topology

Status: **bones only.** This captures the design so we can build it in chunks
later. Nothing here is implemented yet.

## Context

NetGrip's 0.1/0.2 work made the canvas *honest* (every box is freshly probed
reality) and *persistent* (Save writes through the host's backend). 0.3 is two
threads:

1. **The network features** already listed in [ROADMAP.md](../ROADMAP.md) §0.3 —
   bridge creation, non-default routes, DNS search domains, DHCP client control,
   `teamd` read. That's the "big work" and stays roughly as written.
2. **A clarity & terminology pass** (this doc) — the canvas and menus should
   *name kernel objects accurately* and *teach the user the differences*, so a
   VLAN, a macvlan, a veth and a bridge never get blurred into one "virtual"
   blob. This is the layer the network features sit on, so it's worth doing
   first: it's low-risk chrome, and it sets the vocabulary the rest reuses.

The trigger: we nearly collapsed "VLAN" into a generic "virtual interface". They
*share a feel* but are genuinely different kernel objects (see Glossary). The
fix isn't to rename one into another — it's to give the family a shared look
(colour + a "virtual interface" category in the legend) while each member keeps
its correct name and an in-app **Definitions** page explains why.

## Terminology decision

- **Keep the accurate kernel names** on every box: a VLAN stays "VLAN", a bridge
  stays "bridge", a bond stays "bond". We do **not** rebrand VLAN → veth/virtual.
- **"Virtual interface" becomes a *category*, not a box label** — the colour
  bucket the legend names (currently the teal `vlan` colour). VLAN is its only
  member today; macvlan/veth/bridge-vlan join later (ROADMAP §0.4).
- **Internal identifiers stay** (`kind="vlan"`, `VlanNode`, `plan_create_vlan`,
  `vlan_id`, `vlan_parent`). They're the correct kernel term and a rename is
  churn with no user benefit. Only *user-facing strings* change.
- The "New …" creation entry point becomes a small submenu structured to host
  the future types, but in 0.3 only **VLAN (802.1q)** is wired up.

## Glossary (the source of truth for the in-app Definitions page)

Plain-language, framed by how Linux structures each (Windows someday). This text
is the content for the Help → Definitions dialog and the one-line legend hints.

- **Physical interface (NIC)** — a real hardware port (or its driver). **Wired**
  (Ethernet) or **Wireless** (Wi-Fi, 802.11). Kernel: a netdev with no
  `info_kind`; wireless is detectable via `/sys/class/net/<dev>/phy80211`.
- **VLAN subinterface (802.1q)** — a **tagged** child of exactly one parent.
  Frames it sends carry a VLAN tag (id 1–4094); one parent can host many VLANs,
  each its own L3 interface. `ip link add link eth0 name eth0.10 type vlan id 10`.
- **macvlan** — an **untagged** child of one parent with its **own MAC**, sharing
  the parent's wire/segment. Looks like a separate NIC on the same broadcast
  domain (no tagging). vs VLAN: macvlan = new MAC, *same* domain; VLAN = *separate
  tagged* domain. `type macvlan`.
- **veth pair** — a virtual **cable with two ends**; what enters one end exits
  the other. Always created as a pair, typically to join a container's namespace
  to a host bridge. `type veth peer name …`.
- **bridge** — a software **L2 switch**: several ports joined into one broadcast
  domain; learns MACs and forwards between members. A *vlan-aware* bridge can
  also tag/filter per port (PVID + tagged lists). Members are enslaved via
  `master`. `type bridge`.
- **bond / team** — several NICs **aggregated into one logical link** for
  redundancy or throughput (LACP, active-backup, …). Members enslaved to the
  master.
- **loopback (`lo`)** — the host-internal interface (127.0.0.1 / ::1).

How NetGrip buckets these onto colours (the legend):

| Category | Colour key (`theme.py`) | Box subtype line |
|---|---|---|
| Physical interface | `nic` | Wired / Wireless |
| Group (bond/bridge/team) | `group` | bond mode / "bridge" / "team" |
| Virtual interface (VLAN; later macvlan/veth) | `vlan` | tag (VLAN id) / peer |
| IPv4 config | `ip4` / `region4` | — |
| IPv6 config | `ip6` / `region6` | — |
| System DNS | `dns` | — |
| Loopback | `loopback` | loopback |

## Workstreams

Each is independently shippable. Suggested order = low-risk chrome first, so the
vocabulary and view controls land before the network features build on them.

### A. View menu + window chrome rework
The toolbar today carries Host, Refresh, Fit, Auto-layout, **Show loopback**,
Theme; the menubar carries only **Help**. Rework into:

- A new **View** menubar menu holding three checkable, QSettings-persisted items:
  - **Show loopback** (moved off the toolbar; same behaviour as today).
  - **Legend** (workstream B).
  - **Hide offline** (workstream A2).
- **Refresh as an icon** — `QIcon.fromTheme("view-refresh")` (text fallback if
  null), `setToolTip("Refresh (F5)")` so the shortcut shows on hover.
- **Help as a right-aligned `?` button** — push it to the right with an expanding
  spacer `QWidget` in the toolbar; a `QToolButton` (`?`, InstantPopup) opens a
  menu: **About NetGrip** and **Definitions…** (workstream E). Drop the menubar
  Help menu.

Refactor note: `populate(...)` is called ~6 places as
`self.canvas.populate(self.state, self.loopback_action.isChecked())`. Replace
those with one `self._repopulate()` helper that passes the current view options,
and have the canvas hold the options (it already stores `_show_loopback`).

Files: [src/netgrip/ui/main_window.py](../src/netgrip/ui/main_window.py)
(`_build_toolbar` → split into `_build_toolbar` + `_build_menus`),
[src/netgrip/ui/canvas.py](../src/netgrip/ui/canvas.py).

#### A2. Hide offline
When enabled, drop interfaces where `not iface.is_up` from the `shown` list in
`Canvas.populate` (one extra predicate in the existing comprehension at
[canvas.py:119](../src/netgrip/ui/canvas.py#L119)). Their IP groups, members and
edges fall away for free (the layout already guards `in shown_names`). Drafts are
canvas-only, so they're untouched — exactly "show drafts but not offline bridges
or wired interfaces". Loopback keeps its own separate toggle.

### B. Legend overlay
A pinned, toggleable key so boxes can lean on colour instead of spelling out the
category. Mirror the existing top-right corner-widget pattern (the floating Save
button: `Canvas.set_corner_widget` / `position_corner_widget`,
[canvas.py:82](../src/netgrip/ui/canvas.py#L82)) but anchored **top-left**.

- New `src/netgrip/ui/legend.py`: a small `QFrame` of rows = colour swatch +
  category name (the table above). Colours pulled live from `theme.py`; rebuilt
  on theme change (like `save_button.setStyleSheet` is re-applied in
  `_theme_picked`).
- Hidden by default; visibility persisted in `QSettings("legend_visible")`.
  Repositioned in `resizeEvent`.
- Generalize the canvas corner-widget hook to take a corner (top-left vs
  top-right) rather than duplicating it.

Files: new `src/netgrip/ui/legend.py`, `canvas.py`, `main_window.py`,
optionally a small data table in [theme.py](../src/netgrip/ui/theme.py)
enumerating `(category label, colour key, one-line hint)` so the legend and the
Definitions intro share one source.

### C. Box "type" line — Wired / Wireless on physical NICs
Bridges already show "bridge" / bonds show their mode (`GroupNode`,
[items.py:206](../src/netgrip/ui/items.py#L206)); physical NICs show no type.
Add **Wired / Wireless**. This is the "add a field read from the host" recipe:

1. `core/model.py` — add `wireless: bool = False` to `Interface`.
2. `core/probe.py` — detect wireless from sysfs (a single extra runner command,
   e.g. list `/sys/class/net/*/phy80211`), merged onto interfaces by name. Stays
   Qt-free; works locally and over SSH.
3. `core/demo.py` — mark `wlan0` (already present at
   [demo.py:73](../src/netgrip/core/demo.py#L73)) wireless; add the value to the
   `tests/test_probe.py` fixture and assert it parses.
4. `ui/items.py` — `_iface_detail` ([items.py:141](../src/netgrip/ui/items.py#L141))
   adds a "Wired"/"Wireless" line for `kind == "physical"`.

Files: `core/model.py`, `core/probe.py`, `core/demo.py`, `tests/test_probe.py`,
`ui/items.py`.

## How this relates to ROADMAP §0.3 network features
The network features (bridge creation, static routes, DNS search, DHCP control,
team read) are unchanged and remain the "big work". This clarity pass front-runs
them because:
- **Bridge creation** lands cleaner once the legend/glossary already explain what
  a bridge *is* vs a bond and a VLAN.
- The **virtual-interface category + submenu** is the natural home for
  macvlan/veth creation when §0.4 arrives.
