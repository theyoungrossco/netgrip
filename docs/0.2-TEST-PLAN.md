# 0.2 "Persistence" — test plan

The 0.2 work is large, so it lands in milestones that are each independently
testable. This file tracks how to verify each one. Ross tests as the work lands;
update the status column as milestones complete.

The headline addition beyond the roadmap is a **three-way action model** for
every mutation:

| Action | Effect | Reverts? |
|---|---|---|
| **Try** | Apply to the *running* config, then auto-revert after a timeout unless kept | Yes, host-side, even if the connection drops |
| **Apply** | Apply to the running config and leave it (today's behaviour) | No (but a managing backend may revert it on reboot) |
| **Save** | Write persistent config through the detected backend (and apply it) | No — survives reboot |

## M1 — Backend detection + indicator  (✅ ready)

**What changed.** A new Qt-free `core/backends.py` runs one best-effort probe
(`systemctl is-active NetworkManager / systemd-networkd`, `ls /etc/netplan`) and
classifies the host's config owner into `networkmanager | networkd | netplan |
ifupdown | runtime | unknown`. It is attached to `HostState.backend` and shown in a
permanent status-bar indicator (bottom-right): **"Persist: <backend>"**, dim
when a change can be persisted, in the warning colour when it can't.

**Manual tests**

1. **Demo** — `netgrip --demo`.
   - Bottom-right shows **"Persist: netplan"** in the normal/dim colour.
   - Hover it: tooltip reads *"Configured by netplan (1 file in /etc/netplan,
     rendered by systemd-networkd)… Save will write persistent configuration
     through this backend."*
2. **Local host** — launch normally against your own machine.
   - Indicator names your real backend (NetworkManager on most desktops). Cross-
     check: `systemctl is-active NetworkManager systemd-networkd; ls /etc/netplan`.
3. **Runtime-only host** — a container/minimal box with none of the three
   active. Indicator shows **"Persist: Runtime only"** in the *warning colour*,
   tooltip warns changes are lost on reboot.
4. **SSH host** — connect to a remote box; indicator reflects *that* host, and
   updates when you switch hosts. Disconnect → indicator clears.
5. **Unreadable host** — if a probe can't run, indicator shows **Unknown**
   (warning colour) and the app does not error.

**Pass criteria.** Indicator always matches the host actually selected; never
throws; colour distinguishes persistable from runtime-only.

## M2 — `IpGroupDialog` static pre-fill  (✅ ready)

**What changed.** Opening the per-family IPv4/IPv6 settings for an interface
that already has a *static* address now opens with **Static selected and the
address pre-filled**, instead of defaulting to an empty Dynamic field that hid
the current config. A DHCP/RA lease still defaults to Dynamic.

**Manual tests** (`netgrip --demo`, right-click an IPv4/IPv6 region → settings)

1. **bond0 · IPv4** (static `10.0.0.5/24`, no lease) → opens **Static**,
   field shows `10.0.0.5/24`. OK with no change → confirmation has **no address
   command** (it's a no-op for the address).
2. **eth0 · IPv4** (DHCP lease `192.168.1.10/24`) → opens **Dynamic**, greyed
   value `192.168.1.10/24` (unchanged behaviour).
3. **wlan0 · IPv4** (no address) → opens **Dynamic**, empty (unchanged).
4. Editing the pre-filled Static address to a new CIDR and OK → confirmation
   shows `ip address add <new> dev bond0`.

**Pass criteria.** Existing static config is visible/editable; opening then OK
without edits never proposes a destructive command.

## M3 — Try / Apply + Static-replaces-DHCP  (✅ ready)

**What changed.** The command-confirmation dialog now offers **Apply** (today's
runtime change, left in place) and, where a gesture has a safe inverse, **Try**.
Try applies the change *and* arms an automatic revert; a **Keep / Revert now
countdown** then asks whether to keep it. The revert is scheduled **host-side
and detached**, so it still fires if NetGrip is killed or the SSH connection
drops. (Save — the persist action — is the toolbar button arriving with M4.)

Separately, in the IPv4/IPv6 settings dialog, choosing **Static now removes the
DHCP/RA address** of that family instead of stacking a static one on top of it.

Try is now offered on **every gesture with a safe inverse**: add / edit / move /
clone / detach an address, IPv4/IPv6 settings, clear gateway, link up/down, link
properties (MTU / MAC / alias / rename), add / remove a bond member, bond mode,
and create VLAN (menu or draft). Still **Apply-only** (inverse is unsafe or
ambiguous, to revisit): creating a bond, re-parenting a VLAN, and attaching a
draft. A confirm dialog with no Try means that gesture is in the Apply-only set.

**Fixes from first round of testing**

- Reverts that re-add an address use `ip address replace` (idempotent), so a
  DHCP client re-adding a lease mid-Try no longer makes Revert fail with
  "Address already assigned". Multi-step reverts chain with `;` (best-effort),
  not `&&`.
- Choosing **Static with the address you already hold via DHCP** is now a
  no-op at runtime (it used to delete the lease and add nothing back). A
  status-bar note explains it needs Save to pin as static.
- **Add IPv4/IPv6 address** now offers Try (it had only Cancel/Apply).

**Manual tests** (`netgrip --demo` shows the buttons; a real/local host runs
them — demo refuses execution as before)

1. **Confirm dialog** — drag an IP box onto another NIC. The confirm dialog
   shows **Cancel / Try (60s) / Apply**, with a note explaining Try auto-reverts.
2. **Apply** — click Apply: the change lands and stays (today's behaviour).
3. **Try → Keep** (local/real host) — Try the move; the **countdown dialog**
   appears; click **Keep**; the change persists (runtime), countdown gone.
4. **Try → Revert now** — Try again; click **Revert now**; canvas returns to the
   original state immediately.
5. **Try → time out** — Try; wait 60s; it reverts on its own.
6. **Try → connection loss (the safety net, real SSH host)** — Try a change,
   then *kill NetGrip* (or drop the link) before deciding. After ~70s the host
   reverts on its own. Verify with `ip addr`/`ip route` on the host. **This is
   the headline guarantee — test it on something you can afford to cut.**
7. **Static replaces DHCP** — on an interface with a DHCP IPv4 lease, open IPv4
   settings (now opens Dynamic), pick **Static**, enter an address, OK. The
   confirm plan shows `ip address add <static>` **and** `ip address del <lease>`.
   Try it: the lease is gone while tried, restored on revert. (Note: a live DHCP
   client may re-hand the lease — making it stick is M5/Save.)

**Pass criteria.** Try always recovers the original state (Keep is the only path
that retains the change); a dropped connection still reverts host-side; Static
no longer leaves a stale DHCP address behind.

## Local admin-password caching  (✅ ready)

**What changed.** On the local machine, instead of `pkexec` prompting on *every*
action, NetGrip now drives `sudo` with a password collected **once per session**
(cached in memory, fed to sudo through the same askpass helper the SSH path
uses — never on disk or the command line). Order of preference: passwordless
sudo → cached-password sudo (sudoers who must authenticate) → pkexec (e.g. a
non-sudoer with polkit). A wrong password is cleared and re-prompted.

**Manual tests** (local host where sudo needs a password)

1. First privileged action (Apply/Try) → **one** "Administrator (sudo) password"
   dialog. Enter it; the action runs.
2. Do several more actions (e.g. a Try then Revert) → **no further prompts**.
3. Enter a wrong password → action fails with "Incorrect administrator
   password"; the next action prompts again (no loop on the bad password).
4. Passwordless-sudo host → no prompt at all (unchanged). Root → no prompt.
5. SSH host → unaffected (its own password flow; remote sudo stays passwordless).

**Pass criteria.** At most one password prompt per session on a sudo host; the
secret never appears in `ps`/logs (`tests/test_runner.py` asserts it stays in
the askpass env, not the argv).

## M4 — Save toolbar + backend write-through  (✅ ready)

**What changed.** A **Save** toolbar button (after Auto-layout) commits **all
unsaved changes** — runtime changes made via Apply / Try-kept this session — to
persistent config through the detected backend. It shows an unsaved count
(`Save (2)`) and is enabled only when there *are* unsaved links **and** the
backend can write (NetworkManager / networkd / netplan; disabled on
runtime-only). Save persists the **current running IP config** of the touched
links (declarative), not a replay of deltas — so it captures wherever Apply /
Try-keep actually left them, and re-Saving is idempotent.

- **Scope:** IP config only — addresses, gateway, DNS, DHCP-vs-static per family.
  *Not* device topology (bonds/bridges/vlans aren't created persistently — a
  later step). `lo` is never marked unsaved.
- **Dirty-tracking** (`actions.affected_links`) is lexical: a successful Apply or
  Try-*keep* marks the links its plan named; a *reverted* Try marks nothing.
  Switching hosts clears the set.
- **Per backend** (`core/persist.py`, all pure + unit-tested):
  - **networkd** → writes `/etc/systemd/network/10-netgrip-<name>.network`
    (`[Match] Name=` + `[Network] DHCP=/Address=/Gateway=/DNS=`), then
    `networkctl reload` + `reconfigure <name>`.
  - **netplan** → one `/etc/netplan/90-netgrip.yaml` (sectioned by kind, gateway
    as a `routes: default`), `chmod 600`, `netplan apply`. *Experimental* — rely
    on the confirm-dialog review.
  - **ifupdown** (Debian/Proxmox, ifupdown2) → a drop-in
    `/etc/network/interfaces.d/90-netgrip.cfg` (`auto` + `iface … inet/inet6
    static|dhcp`, CIDR `address` lines), then `ifreload -a`. The stock
    `source /etc/network/interfaces.d/*` line pulls it in, so it *adds to* the
    host's `/etc/network/interfaces` rather than rewriting it. *Experimental.*
  - **NetworkManager** → resolves device→connection (`nmcli -t -f DEVICE,NAME
    con show`), then `nmcli con mod <con> ipv4.method/addresses/gateway/dns …`
    (+ ipv6) and `nmcli con up <con>`.
- Save reuses the normal confirm → escalation → run → re-probe path
  (**Apply-only**: persisting *is* the commit, so no Try). The file-write step is
  pretty-printed in the confirm dialog as `# write <path>:` + the indented body
  (M4d), not a raw heredoc blob.

**Fixes from first round of testing**

- The **Save** control is now a floating button pinned to the canvas's
  top-right, in the attention/red colour, **hidden until a Save is possible**
  (unsaved links *and* a writable backend) — far more visible than the old
  toolbar entry. `Ctrl+S` triggers it while shown.
- **No more manual refresh.** A change can finish landing after its command
  returns (a link coming up gets a DHCP/RA lease; a Save's `nmcli con up` /
  `netplan apply` re-activates the link while IPv6 re-acquires via RA), and a
  single re-probe could catch it mid-flight and sit on the wrong state. Now every
  Apply/Save schedules follow-up re-probes (Save: after the settle *and* again
  ~6 s later), **and** a gentle background poll (~10 s) keeps the canvas current
  on its own — including changes made outside NetGrip. The poll self-guards: it
  skips while a probe/apply is running, while the mouse button is down
  (mid-drag), and while any dialog or context menu is open, so it never
  interrupts a gesture or redraws under a dialog.
- **A family no longer loses its box when its last address is deleted *if* it
  still has a gateway or DNS** (`Interface.configured_families`) — otherwise that
  config was orphaned and unreachable. A genuinely empty family (no address,
  gateway or DNS) still draws no box; re-add via the link's "Add IPv4 config…".
- **Adding a config to an address-less family now works:** the addressing toggle
  defaults to **Static** (editable) instead of Dynamic. Dynamic is a runtime
  no-op today (starting DHCP is a later backend), so the old default made OK
  apply nothing. A family with a live DHCP/RA lease still defaults to Dynamic.
- **NetworkManager: switching to DHCP no longer resurrects the static.** Setting
  `ipv4.method auto` alone leaves the profile's `ipv4.addresses` in place, and NM
  re-applies that static on `con up` — so a deleted/Dynamic'd static kept coming
  back. Save now sets each configured family's method **and** its
  addresses/gateway/DNS explicitly, blanking them (`ipv4.addresses ""`, etc.) when
  empty, so the profile matches the intended config. An unconfigured family is
  left untouched.
- **ifupdown backend added.** A Debian/Proxmox host that keeps its config in
  `/etc/network/interfaces` was previously detected as **Runtime only** (Save
  disabled, nothing persisted — which is why `interfaces.d` stayed empty). It is
  now detected as **ifupdown** and Save writes the drop-in above. Verify: the
  indicator reads **"Persist: ifupdown"**; after a Save, `cat
  /etc/network/interfaces.d/90-netgrip.cfg` shows the stanza and `reboot` keeps
  it. **Review the confirm dialog** — a wrong stanza + `ifreload -a` can drop a
  remote link, so Try-test risky changes first.

**Manual tests** (`netgrip --demo` shows the button/plan; a real host runs it)

1. **Disabled by default** — fresh launch: **Save** is greyed (nothing unsaved).
   On a runtime-only host it stays greyed even after a change (nowhere to write).
2. **Count tracks changes** — Apply a change (e.g. add an address, change MTU):
   the button becomes **Save (1)**. A second change on another link → **Save (2)**.
   A Try you **Revert** must *not* bump the count; a Try you **Keep** must.
3. **Demo review** — with an unsaved link, click **Save**: the confirm dialog
   shows the write step as `# write /etc/netplan/90-netgrip.yaml:` followed by the
   readable YAML, then `chmod 600 …` and `netplan apply`. Demo refuses to run it.
4. **networkd host (real)** — make a change, Save, confirm. Check
   `/etc/systemd/network/10-netgrip-<name>.network` exists with the right
   `[Network]` block; `reboot` and confirm the address/gateway survive.
5. **netplan host (real)** — Save, confirm; `/etc/netplan/90-netgrip.yaml` is
   written `600`, `netplan apply` runs, config survives reboot. **Review the YAML
   in the dialog first** — netplan re-renders the whole stack.
6. **NetworkManager host (real, e.g. a desktop)** — Save; the plan is `nmcli con
   mod …`/`con up …` against the link's *active connection*. `nmcli con show
   <con>` reflects it; reboot keeps it. A link with no active connection Saves
   with a clear error, not a crash.
7. **Count clears on success** — after a successful Save the button returns to
   **Save** (0). Switching hosts also clears any pending count.

**Pass criteria.** Save is enabled only when there is something persistable;
the confirm dialog shows reviewable file contents; a Save survives `reboot` on
each managed backend; the unsaved count is honest (Try-revert never dirties).

- **Remote-server safety:** Try first, or Save with care — a wrong *persisted*
  gateway/address that also drops the link is the dangerous case, since it now
  comes back on reboot too.

## M5 — Dynamic addressing actionable  (✅ ready)

**What changed.** In IPv4/IPv6 settings, choosing **Dynamic** for a family that
is currently *static* now records a pending **"→ DHCP"** switch instead of doing
nothing. It is declarative, applied at **Save**: NetGrip does *not* run a runtime
`ip` command (which would drop the address with nothing to replace it) — the
static stays on the link until Save writes `dhcp` through the backend and its
reload performs the swap (e.g. ifupdown2's `ifreload -a`, `netplan apply`,
`networkctl reconfigure`, `nmcli … ipv4.method auto`).

- The family's box stays visible and gains a **"→ DHCP on Save"** line while the
  switch is pending; the Save count includes it. Choosing **Static** again (or
  re-adding a static address) cancels the pending switch.
- `LinkConfig.set_dhcp(family)` drops that family's static address, gateway and
  DNS and flags the lease, so every backend renderer emits the DHCP form. The
  pending marker auto-clears after Save once the re-probe shows the static gone.

**Manual tests** (`netgrip --demo` to see the marker/plan; a real host applies it)

1. **Switch registers** — on a link with a static IPv4, open IPv4 settings, pick
   **Dynamic**, OK. The box keeps its address but shows **"→ DHCP on Save"**, and
   the Save button appears/increments. (Previously this did nothing.)
2. **Save writes DHCP** — click Save: the confirm dialog's file shows `dhcp4:
   true` / `iface … inet dhcp` / `DHCP=ipv4` (per backend) with the static gone.
3. **Real host** — after Save + reboot the link comes up on DHCP; the static is
   not reapplied. (If the static still returns, check it isn't *also* defined in
   the main `/etc/network/interfaces` — a NetGrip drop-in can't override a stanza
   the base file already owns.)
4. **Cancel** — switch to Dynamic, then re-open and pick **Static** with an
   address: the pending marker clears and no DHCP is saved.
5. **Regression (M2)** — Dynamic on a family that is *already* DHCP (a live lease,
   no static) records nothing and proposes no change.

**Pass criteria.** Switching a static family to Dynamic is a visible, saveable
change; Save persists DHCP and reboot keeps it; an already-dynamic family stays a
no-op.

## M6 — Persistent link properties (rename / alias / MAC / MTU)  (✅ ready)

**What changed.** Link-layer properties — a rename, the ifalias, the MAC and the
MTU — now persist on Save, the last of the 0.2 roadmap's persistence items.
Before this they were marked unsaved but **silently dropped** at Save (which only
ever wrote IP config), so the count cleared while nothing was persisted.

They take a different path than IP config: a systemd **`.link`** file
(`core/persist_link.py`), processed by `systemd-udevd` *beneath* whichever backend
owns addressing, so one mechanism persists them on every managed host (all run
systemd). Save writes `/etc/systemd/network/10-netgrip-<name>.link` with a
`[Match] OriginalName=` (the device's boot name — the *pre-rename* name for a
renamed link) and a `[Link]` section carrying **only the properties you changed**
(`Name=` / `Alias=` / `MACAddress=` / `MTUBytes=`), then `udevadm control
--reload`.

- **Dirty-tracking is per property.** Editing MTU/MAC/alias/name in a NIC's
  Properties marks the link unsaved *and* records which properties changed, so
  the `.link` file pins only those (never a MAC you didn't set). Apply or
  Try-**Keep** records it; a Try you **Revert** records nothing. A rename moves
  the dirty entry to the new name and remembers the boot name for the match.
- **It rides the same Save.** A link with both an IP change and an MTU change
  Saves both — the backend's IP write-through (`nmcli` / `.network` / YAML /
  `interfaces.d`) **and** the `.link` file — in one confirm → run → re-probe.
- **Applies on next boot, deliberately.** NetGrip already made the runtime
  change; Save does **not** re-trigger udev (renaming a live link needs it down,
  and a forced device re-add could drop a remote link). The file makes it survive
  reboot; the running state already matches it.

**Manual tests** (`netgrip --demo` shows the plan; a real host runs it)

1. **MTU only** — Properties on a NIC, change MTU, Apply. Save appears (count 1).
   Click Save: the confirm dialog shows `# write
   /etc/systemd/network/10-netgrip-<name>.link:` with `[Match] OriginalName=<name>`
   and `[Link] MTUBytes=<n>` and **no** `Name=`/`MACAddress=`/`Alias=` lines.
2. **Rename** — rename `enp3s0`→`lan0`, Apply (the link is now `lan0`). Save: the
   file is `10-netgrip-lan0.link`, `OriginalName=enp3s0`, `Name=lan0`. The Save
   count is 1 — the stale old name doesn't linger.
3. **Alias clear** — set an alias and Save, then clear it and Save again: the
   second file has a bare `Alias=` line (clears the ifalias).
4. **Combined with IP** — on one link change both the MTU and the IPv4 address,
   Apply both. Save (count 1) shows the backend IP write **and** the `.link`
   write in the same plan.
5. **Try-Revert doesn't dirty** — Try an MTU change, Revert now: Save must not
   appear for it.
6. **Real host + reboot** — Save an MTU/alias/rename, `cat` the `.link` file,
   `reboot`: the property survives (a rename: the device comes up as `lan0`).
7. **NetworkManager host** — the `.link` file is written *in addition* to the
   `nmcli` IP commands. NM may also manage MTU/MAC via its own profile, so review
   the dialog — this is the experimental, universal-udev path.

**Pass criteria.** A link-layer change is a visible, saveable change that writes a
reviewable `.link` file containing only the edited properties; Save no longer
silently drops MTU/MAC/alias/rename (the prior gap); the change survives reboot.
